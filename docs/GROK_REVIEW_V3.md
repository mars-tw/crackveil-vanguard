### 1.1 F12 / beauty 模式 — release 安全審（重點）

#### 啟用路徑與門檻（現況碼）

| 入口 | 條件 | 行為 |
|------|------|------|
| `KEY_F12`（`hud._unhandled_input`） | `event.pressed` 且 **`OS.is_debug_build()`** | toggle `set_screenshot_beauty_mode` |
| CLI `--screenshot-beauty`（`hud._ready`） | **`OS.is_debug_build()`** 且 cmdline 含 flag | 開機即 ON |
| `HUD.set_screenshot_beauty_mode(true)` | **`if enabled and not OS.is_debug_build(): return`** | release 無法寫入 ON |
| `EntityFactory.get_visual_composite_layer_count` | **`OS.is_debug_build() and GameManager.screenshot_beauty_mode`** 才 `return 4` | 即使 flag 被污染，release 仍走 adaptive LOD |
| 預設 | `GameManager.screenshot_beauty_mode = false`；`screenshot_beauty_active = false` | 冷啟動關閉 |

#### 裁決：release 真的不啟用嗎？

| 檢查 | 結果 |
|------|------|
| 玩家在 **export-release**（含現有 Web preset `export-release`）按 F12 | **無效** — 輸入分支根本不進 |
| 玩家對 release 建置加 `--screenshot-beauty` | **無效** — `_ready` 雙條件失敗 |
| 理論上外部腳本直寫 `GameManager.screenshot_beauty_mode = true` | **仍無 4 層** — EntityFactory 再驗 `is_debug_build()` |
| 正式路徑（傷害、spawn、敵數、Meta、設定）是否讀 beauty | **未見** — 僅 composite layer 決策 |
| Stress 是否被 beauty 綁架 | **否** — Stress 直接問 `MOBILE_TUNING.vfx_composite_layer_count`；beauty 預設關 |

**§1.1 裁決**：**成立 — release 不啟用**。  
這不是「文件寫了就算」：是 **輸入門 × API 門 × LOD 消費門** 三層（加預設 false 共四道）。  
公開 Web（`export/web/index.pck` = 4,659,392 bytes，HEAD = `152237f`）屬 release 語意，玩家拿不到 beauty 提層。

#### 品質／契約保留意見（非阻擋）

| 項 | 嚴重度 | 說明 |
|----|--------|------|
| R14 **未** assert `adaptive_layers == 3` | 測試債 | 只印出；能擋 beauty=4，擋不住 adaptive 被默默改 4 |
| R14 **未**模擬 release／`not is_debug_build` | 測試債 | 三層門檻靠靜態審；未來可加「強制假 debug=false 的純函式契約」若重構 |
| Beauty **無凍幀** | 相對 V2 建議 C5 | V2 提案含 0.15s 凍幀；落地僅「藏 HUD + 新生 4 層」→ **部分成立**，夠用 |
| Beauty 藏 **整個 `root`** | 工作流 | 搖桿正確藏（`virtual_joystick` 是 `root` 子節點）；**Phase II 橫幅也一起消失** → 要「有 stakes 字」的 L1 必須 **beauty OFF 或先截橫幅再截淨畫面** |
| 只影響 **切換後新生** VFX | 已文件化 | 正確；拍煙尾須先 ON 再引爆，勿期望舊 burst 補層 |
| `print("SCREENSHOT_BEAUTY_…")` | P2 | debug 噪音可接受；release 路徑碰不到 |
