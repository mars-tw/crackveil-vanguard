## 2. Frame 2 目標重驗：exploit 與體感風險

### 2.1 取捨本質（把契約改寫了）

設計初稿心智模型：

> CD 觸發 → spatial 找最近敵 → 沿主人→目標投放。

實作攻擊管線：

> 0–1 蓄力鎖定語意 → **frame 2 才重驗並投放**；活目標出距＝揮空；**原目標死亡或 pool token 換代＝重取最近敵**。

報告自承：滿編時「其他隊員常在兩個預備幀內擊殺鎖定目標」。  
→ **重取不是邊緣 case，是滿編常態路徑**。武器實際契約變成：

> **「命中幀的最近可投放目標」**，蓄力幀鎖定多半是表演／可廢棄狀態。

這與「anticipation → impact → recovery」動畫契約相容，但與「玩家讀蓄力方向＝落地點」的操作可讀性可能衝突。

### 2.2 體感風險

| 風險 | 說明 | 嚴重度 |
|------|------|--------|
| **動畫指向 ≠ 投放方向** | 朝 A 抬燈籠，A 被隊友秒掉，frame 2 轉向 B；玩家看到「砍空氣卻在另一側長出裂傀」 | 高（信任感） |
| **編成依賴的命中率** | 滿編幾乎不揮空；單牧／低輸出窗大量出距揮空 → 同武器體感隨 roster 跳變 | 高 |
| **承諾懲罰不對稱** | 敵人**跑出 range** 懲罰揮空；敵人**被殺**給免費重瞄 — 玩家直觀是「都沒打到原目標」，獎懲卻相反 | 中高 |
| **走位微操被稀釋** | 設計要用走位決定覆蓋；若常態是 impact 瞬間最近敵，玩家更傾向「站樁等 CD + 讓隊友清鎖」而非預判落點 | 中 |
| **雙裂傀 L2** | 同 tick 兩具若共用同一重取快照還好；若兩次 query 中間插入 FIFO／死亡，可能並肩方向不一致或第二具吃到不同最近敵 | 中（邊界） |

### 2.3 Exploit／優化玩法（非外掛，是系統可被學歪）

1. **「殺steal 保險投放」**  
   滿編清雜快時，牧者 CD 幾乎總能在 impact 吃到新最近敵 → **有效命中率被隊伍 DPS 補貼**。牧者強度隨「隊內搶人頭速度」上升，BalanceMock 若只測單英雄或固定編，會**低估滿編強度**。

2. **蓄力中位移騙落點**  
   若生成點用「impact 時主人位置 + 重取方向」，玩家可在 frame 0–1 强行位移，等隊友擊殺原目標後，把裂傀錨在**新路線**上。這不是 bug，但把「固定站位要先佈陣」變成「動畫中再微調錨點」，**削弱 move_speed 218 的設計意圖（先佈再動）**。

3. **Pool token 換代＝重取**  
   若 token 換代判定過寬：敵人物件回收再利用時，可能把「還是合法仇恨對象」誤判成需重取，導致**更常跳向當幀最優目標**（變相智慧鎖定）。  
   若過嚴：反之假揮空。報告無邊界測例。

4. **揮空仍走滿 recovery**  
   正確且防連點 cancel；但在「出距揮空多、死亡重取多」混打時，玩家難以從手感分辨失敗原因（跑掉 vs 邏輯重取成功），**教學成本**上升。

5. **與「真正揮空零生成」文案並存的信任問題**  
   QA／玩家若理解「沒打到就不生成」，看到隊友擊殺後仍生成，會質疑判定；若理解「只有出距揮空」，則 OK。**文件必須寫死失敗條件表**，否則支援與平衡都會吵。

### 2.4 監工建議（審查意見，非改碼）

- 接受重取的前提：必須把失敗條件改成公開契約，例如：  
  - **Whiff**：impact 時無任何 range 內敵人；或原目標仍存活但出距且**禁止**重取。  
  - **Retarget**：僅原目標 `is_dead`／invalid token。  
- 若維持現狀，Balance 必須用**滿編高殺速**與**寡編低殺速**兩檔校 DPS，禁止只信 Stress 的 `whiffs=0`（那可能只證明壓測編太肥）。
- 雙生 L2：強制同一 impact 快照，禁止第二具二次 nearest。

**§2 總評**：取捨工程上自洽，**滿編可用**；但把召喚武從「承諾落點」推向「智慧落點」，**有體感分裂與編成補貼型強度**，且報告用 `whiffs=0` 當好事，監工視為**可能掩蓋常態重取**。

## 3. 羈絆重算時機遺漏（換武器／進化／質變）

### 3.1 規格只掛三鉤子

設計 §3.2／實作：

> `recruit` / `death` / `start_squad`（事件驅動，非每幀）

這對「誰在場上」是夠的；對「**武器實例何時重建、cap／倍率何時寫入**」不夠。

### 3.2 高風險遺漏場景

| 場景 | 為何會壞 | 可能症狀 |
|------|----------|----------|
| **質變升級 `construct_anchor` L1→L2** | 武器行為從單投放改雙投放；若 cap／bond 修飾在 recruit 時 push 進 instance cache，質變後可能仍用舊 `max_live` 或未吃 `×0.85` | 雙生卻 cap 仍當 3；或傷害沒打折 |
| **進化 `evo_mirror_flock`** | 設計：`projectile_count` 語意 +1、`hit_interval×0.9`、碎裂波；**bond 不重算** | 進化後同時上限仍停在進化前；或碎裂波不吃隊長／羈絆增傷（視讀取點） |
| **`bond_captain_shepherd` 的 cap +1 與進化 +1 疊加** | 正確應是 `min(base + evo + bond, 6)` **在每次 spawn／FIFO 判定時拉取** | 只在 recruit 算一次 → 進化後 cap 錯誤；或 bond 斷了 cap 不降 |
| **隊長或牧者死亡 → 羈絆掉，但場上已有 6 具** | 設計：立即失效、不回溯傷害；**沒寫是否立刻 FIFO 砍到新 cap** | 幽靈多 1 具直到自然到期＝死亡後仍多吃數秒 cap 紅利 |
| **`bond_ember_pulse` / `bond_void_rail` 掛在行為鍵倍率** | 擲彈／脈衝／網／狙若在質變／進化時 **rebuild WeaponRuntime from Resource** | 進化後失去 +8% 半徑、+12% 對減速傷、網 +0.4s，直到下次 recruit 奇蹟重算（通常不會） |
| **`bond_guard_echo` 全隊 −5%** | 若以「啟用時遍歷成員寫入 `damage_taken_mul`」 | **後招進的隊員沒減傷**；或羈絆斷了減傷 **黏住**（sticky buff） |
| **歌者／護衛死亡順序** | 需對稱移除 heal mul 與 team DR | 單向只加不減 → 一局內減傷／治療只升不降 |
| **換武器（若隊長槽或升級替換行為）** | 本專案隊長三武固定，跟隨者通常一武；但**質變／進化等同換行為模組** | 凡「換行為」都應視為換武器 |
| **Debug／重開跑、`start_squad` 以外的重置** | 只寫 start_squad | 開發期或 rerun 旗標殘留 |
| **不存在但易誤實作：每幀重算** | 報告說事件驅動 | 若有人「修 sticky」改成 per-frame，違反輕量原則 — 需守住 |

### 3.3 正確架構應長怎樣（審查標準）

羈絆系統有兩種合法實作，報告**沒說用哪一種**：

- **Pull 模型（推薦）**：`recompute_bonds` 只寫 `active_bond_ids`／flag table；武器每次 tick／cast **讀 flag**。則質變／進化**不必**重算羈絆，但 **cap 公式必須在 spawn 路徑即時算**。  
- **Push 模型**：重算時改寫每個武器 instance。則**必須**在 qualitative apply、evolution apply、weapon rebuild 再呼叫 `recompute_bonds()`，否則必壞。

對抗結論：以目前文案「僅招募／死亡重算」推斷，**若實作為 Push，質變／進化是明確漏洞**；若為 Pull，**仍缺「cap 下降時是否回收超量裂傀」與「隊員中途加入是否吃到全隊 DR」的規格**。  
**建議必補測（邏輯，不單效能）**：

1. 先開牧長裂約 → 進化 → 斷言 live cap 語意與 `min(...,6)`。  
2. 進化後打死牧者或隊長 → 羈絆關閉 → 下一 tick 是否仍能維持 6、增傷是否從 16% 回到 0／10%。  
3. 先有護衛+歌者 → 再招第三人 → 第三人受傷是否 −5%。  
4. 斷星盾和聲後，治療與減傷是否立刻回到基線。  
5. 燼脈聯爆開啟後，分別升級 grenade／pulse 質變與進化，半徑／tick 傷是否仍帶 +8%／+6%。

**§3 總評**：**成員進出**有掛鉤；**成長管線（質變／進化／rebuild）與 cap 動態失效**是最大邏輯空洞，比「少一組羈絆」更嚴重。

## 4. 最該補測的三個效能場景

報告已有：一般 Stress、`group_scans=0`、pool 三零、約 8.24 queries/frame。  
**不足**：未全進化、cast 僅 4 次量級、wall-time 不可比。下列三項是「最可能打穿新增成本」的形狀。

### 場景 A｜雙生 L2 + 進化碎裂 + hard cap 6 的 FIFO 風暴（最高優先）

- **編成**：9 人含牧者；`construct_anchor` 2；`evo_mirror_flock`；`bond_captain_shepherd` 可開可關各一檔。  
- **戰場**：高密度小怪長時間貼臉，讓 6 具常駐且 **CD 連續雙投放** → 每 2.4s×0.9… 強制 FIFO 回收最舊 → **碎裂波密集**。  
- **量**：`construct_spatial_queries/frame`、ExplosionArea active／reject、FIFO／shatter 次數、frame p95、pool 三零。  
- **為何**：Steady-state 6 具 tick 是一回事；**「雙生推 FIFO + 碎裂爆炸 cap」是新增尖峰**，現有 Stress（4 cast）幾乎沒打到。

### 場景 B｜裂傀 6 具 tick × 全隊武器同屏的 spatial 總預算（對照 R19 基線）

- **同 seed／同機關機條件**，A/B：擠掉牧者的舊 9 人 vs 含牧者滿裂傀。  
- **鎖**：queries_per_frame、總 spatial query、CPU frame time p50/p95；**禁止**在 1.05 GHz 降頻髒環境宣告不惡化。  
- **為何**：設計允許既有低於 60fps 債務，但要求**新增不得明顯惡化 >10%** — 這是唯一能結案的測法。報告自己說舊英雄 A/B 都跑不完，**代表連對照實驗都失敗，不是牧者已證明無害**。

### 場景 C｜碎裂波與既有爆炸系搶 `ExplosionArea` cap（正確性＋效能雙棲）

- **編成**：牧者進化 + `ember_grenadier` + `pulse_artificer`（可加燼脈聯爆）。  
- **誘發**：多具裂傀同時到期（對齊 lifetime）+ 雙 AoE 高潮。  
- **量**：shatter 擬似傷害次數 vs 預期、explosion reject／drop 率、卡頓尖峰。  
- **為何**：設計用复用爆炸 cap **防實體爆炸**；但若 cap 滿，**碎裂傷害被靜默丟掉**＝玩法被視覺 budget 改掉，**直接撞 §4.4.8／「視覺 budget 不改玩法」**。這比平均 fps 更能抓到「假優化」。

**次選（未進前三但要排期）**：星環協同 B 對裂傀的掃中檢測是否額外 group／線性掃；Web 低階機上 atlas 變大後的貼圖頻寬（PCK 只證體積不證 runtime）。

## 5. 平衡疑慮與數值論據

### 5.1 用設計初值做的「上限信封」估算（非實測）

單裂傀理論天花板（永遠 2 目標、無空轉）：

\[
\text{DPS}_{\text{one}} \approx \frac{\text{damage}}{\text{hit_interval}} \times \text{max_targets}
= \frac{7}{0.55}\times 2 \approx 25.5
\]

未進化、lifetime 5.5 / CD 2.4 → 平均約 **2.3** 具（忽略 FIFO 與空窗）→ 約 **59** 名義 DPS（理想覆蓋）。  
這對「跟隨者 8–12% 隊總 DPS」是否合理，**完全取決於隊總 DPS 標尺**；R19 只給 `leader_dps_share≈0.41`，**沒給絕對 DPS**，故 8–12% **無法從報告驗證**。

質變 L2：同 tick 2 具、單具 ×0.85 → 單次投放名義輸出 **1.7×** 單投放，且場上到達 cap 的速度接近翻倍。  
進化：interval ×0.9 → 單具頻率 +11%；+群牧 12%；碎裂 `0.55×damage` r70 在 FIFO 風暴下會變成**額外週期 AoE 稅**。  
隊長／羈絆：近距 +16%（替代 +10%），再加星環 +15% 攻速（若常駐刷）：

\[
1.16 \times \frac{1}{1-0.15\text{ 若攻速直接縮 interval}} \quad\text{（實作若為 atkspd 而非 interval，需另換算）}
\]

**樂觀滿配信封**（6 具、增傷與攻速、高升 damage）：名義上很容易逼近或超過「單一跟隨者 < 20% 隊總」— **尤其在滿編殺速補貼 frame 2 重取、近乎零揮空時**。  
**悲觀實戰**：固定站位、Boss 移動、覆蓋重疊、max 2 targets、敵人出圈 → 可能遠低於信封，變成「召喚看起來忙、DPS 不佔比」。  
→ **雙峰不確定性極大，沒有 BalanceMock 就是沒平衡。**

### 5.2 具體疑慮（依嚴重性）

1. **FIFO + 進化碎裂 = 意外的「強制換血 AoE」**  
   高 CD 縮短後，為了維持 6 具不斷砍最舊 → 碎裂變成**準週期清雜**。設計說進化後清雜仍弱於雙 AoE 羈絆，但**未給數值證明**；若 shatter 不吃 explosion 傷害衰減、且 r70 常蓋住潮汐，可能**高於**預期的「告別一擊」。

2. **增傷 soft cap 已被設計組合本身威脅**  
   +16%（牧長）+12%（群牧）已 ≥ +25% 若加總；再加星環攻速是**第二軸成長**。§4.3 寫了 cap，實作報告未證明 clamp。

3. **減傷層：小但可堆在正確坦克位**  
   牧者自身 −8%（要 4 具）+ 星盾 −5% + 歌者治療 +10%（羈絆）→ 鼓勵「護衛+歌者+牧者+隊長」生存核。不是無敵（仍 <15% DR soft），但會拉長戰鬥時間，**間接放大固定站位 DPS 總量**（存活越久，地盤輸出積分越高）。

4. **不可被擊殺的裂傀（§7 建議 v1）**  
   敵人無反制，只有「走開」。搭配虛空網窗口，等於**免費的安全 DPS 家具**。策略上正確（降複雜度），平衡上要把傷害壓低 — 又回到「沒測份額」。

5. **10 選 9 的牙齒可能被「永遠踢線紋」化掉**  
   Stress 直接以牧者替換 `line_mender`。若實戰最優也總是踢修補者，則「永遠有一人被擠」變成「永遠擠同一人」，**編隊策略深度假象**。需看招募權重與無牧者通關曲線；報告無。

6. **雙 AoE 羈絆半徑 +8%**  
   面積比 ≈ \(1.08^2 = 1.1664\)（+16.6% 覆蓋），不是「小感覺 +8%」。燃區 tick +6% 再乘在持續傷害上。設計已警覺「清圖過快」，但**無 90s min HP／clear speed 數據**。

7. **縫獵 +12% 對減速**  
   在網持續 +0.4s 後，窗口更長；若減速 status 覆蓋率高，狙擊穩定性上升。屬可接受的 combo，但與牧者無直接關係；**羈絆系統整體可能比牧者本人更動 meta**。

8. **leader_dps_share 目標 0.38–0.48**  
   多一個跟隨者輸出軸（還可能進化 AoE）→ share 下壓風險；若裂傀吃傷少、隊長仍 carry，則 share 仍高。**兩種失敗模式都有，報告零數據。**

9. **`weapon_projectiles` → lifetime +0.4 的映射**  
   正確避開 cap 崩壞；但玩家若期待「更多裂傀」卻只拿到「更久」，**體感獎勵可能偏弱或偏強**（更久＝更接近硬 cap 常駐）。需看升級卡文案是否誠實。

### 5.3 平衡論據不足清單（交付阻擋級）

- 無 `leader_dps_share` / 跟隨者分攤。  
- 無含／不含牧者的通關曲線、隊長死亡率、90s 前清潮。  
- 無揮空率分層（滿編 vs 寡編）。  
- 無 soft cap 實測（受傷 mul、單武增傷 mul）。  
- 無進化後碎裂傷害佔比。  

**沒有這些，任何「數值完全依設計案」只是「初值有貼上」，不是「平衡完成」。**
