# Crackveil Vanguard — 對抗性設計／程式審查 R4

**審查者**：資深遊戲設計師＋Godot 4 技術總監（對抗式，以程式碼為準，不採信 README 語氣）  
**審查對象**：正式可玩版（squad-survivors / 2D 俯視割草 Roguelite）  
**線上**：https://mars-tw.github.io/crackveil-vanguard/  
**對照**：`docs/GROK_REVIEW_R1~R3.md`、`docs/CODEX_RESPONSE_R1~R3.md`  
**方法**：靜態讀碼盤點 `scripts/`、`scenes/`、`resources/`、`project.godot`；本輪**只審不改**，未重跑 Godot headless / 實機長測  
**日期**：2026-07-10  

---

## 執行摘要

| 面向 | 判定 |
|------|------|
| 技術底座（池化／spatial／cap 解耦） | **可玩級可用**；R1–R3 主契約大致維持；殘留為擴充與效能品質債 |
| 單局可玩性（移動→自動攻擊→升級） | **成立**：有小隊、四種武器行為、三敵種、三選一、結算重開 |
| 作為「想一直玩」的 Roguelite | **明顯不足**：升級多為線性數值、敵人只是血速傷差、無 Boss/事件/勝場、金幣無用途、無 Meta |
| 本局差異感（run variance） | **弱**：開局編制固定；差異幾乎只靠「抽到哪個升級／是否招募」 |
| 抉擇張力 | **中前期尚可**（招募 vs 武器），中後期被「同質數值卡」稀釋 |
| 局間回饋 | **接近零**：重開＝整場歸零；金幣只是結算數字 |
| R4 總判定 | **可發布的割草原型**，尚不是有黏著力的 Roguelite；下一版應優先 **質變升級＋敵行為＋生存曲線／Boss 節奏**，再談 Meta |

**一句話**：引擎與戰鬥骨架已經「能撐」，但 **build 質變、威脅質變、進程閉環** 三根 Roguelite 支柱幾乎都還沒立起來——玩家第二局與第十局的體驗高度同構。

優先級（本輪定義）：

| 等級 | 意義 |
|------|------|
| **P0** | 立刻拉高「想再開一局」的快贏：質變、敵行為、Boss/精英、曲線 |
| **P1** | 進程深度：Meta、武器進化、詞綴／祝福、經濟閉環 |
| **P2** | 延展內容、演出、長期營運向系統 |
| **Bug／風險** | 正確性、Web 單執行緒效能、擴充地雷（與 R1–R3 銜接） |

## (1) 現況盤點（以程式為準）

### 1.1 專案骨架

| 項目 | 實況 |
|------|------|
| 引擎 | Godot 4.x（`project.godot` features 寫 4.2；README／匯出為 4.7 Web 單執行緒） |
| 主場景 | `res://scenes/arena/Arena.tscn` |
| Autoload | `GameManager`、`EntityFactory` |
| 渲染 | `gl_compatibility`（Web 友好） |
| 規模 | 約 70+ 腳本／資源場景；GDScript 約四千行量級 |

### 1.2 玩家／小隊／武器

**開局編制**（`resources/squads/default_squad.tres`）：

| 角色 | ID | 武器 | HP | 速度 | passive（資源有寫） |
|------|-----|------|-----|------|---------------------|
| 裂隙隊長（隊長） | `rift_captain` | `riftline_emitter` 直線彈 | 110 | 230 | `leader`（僅敘述相機／失敗中心） |
| 星環護衛 | `orbit_guard` | `orbit_blades` 環繞刃 | 88 | 218 | `guard` **預留未實作** |
| 裂弧斥候 | `arc_scout` | `arc_chain` 連鎖雷 | 82 | 242 | `scout` **預留未實作** |

**可招募**（同檔 `available_heroes`，上限 `max_members = 5`）：

| 角色 | 武器 | passive |
|------|------|---------|
| 脈衝工匠 `pulse_artificer` | `pulse_bloom` 範圍爆 | `blast` **預留** |
| 線紋修補者 `line_mender` | 第二把 `riftline_emitter` | `support` **預留** |

**武器行為種類（4）** — 各有 scene + `WeaponData.behavior_id`：

| 武器 | 行為 | 基數摘要 | 升級軸（`WeaponData.apply_upgrade`） |
|------|------|----------|--------------------------------------|
| 裂線發射器 | linear | dmg 12 / CD 0.86 / 1 發 | 傷、CD、發數+穿透 |
| 星環飛刀 | orbit | dmg 7 / 2 刃 / 穿透 3 | 傷、**CD 無意義**（CD=0 且 multiplier=1）、刃數+穿透 |
| 脈衝爆花 | explosion | dmg 18 / CD 2.25 / r=82 | 傷、CD、半徑 |
| 裂弧雷鏈 | chain | dmg 16 / CD 1.75 / 4 跳 | 傷、CD、跳數+跳距 |

**重要結構事實**：

- 武器**綁在英雄身上**，開局／招募時 `unlock_weapon(starting_weapon_ids)`；**升級池沒有「解鎖新武器類型」**（catalog 的 `available_weapons` 僅供查表）。
- 同一英雄多武器理論上 API 支援，但資料層每人一把。
- `HeroData.passive_*` **全專案無消費端**——角色差異 = 數值皮＋武器皮，不是被動機制。

### 1.3 升級三選一（升級池）

**基礎玩家池**（`GameManager.PLAYER_UPGRADE_POOL`，固定 3 種，**只套在隊長**）：

| id | 名稱 | 效果 |
|----|------|------|
| `move_speed` | 疾步校準 | +20 移動速度 |
| `max_hp` | 裂隙護甲 | +20 最大 HP 並回 20 |
| `pickup_radius` | 回收磁場 | +24 拾取範圍 |

**小隊擴充池**（`squad_manager.build_upgrade_pool`）：

1. **招募卡**（`recruit_hero`）：尚未入隊、未死過、未滿編的英雄。  
2. **每位在場英雄 × 每把武器 × 3 種**：`weapon_damage` / `weapon_cooldown` / `weapon_projectiles`。

**開局池大小估算**：

- 個人 3 + 可招 2 + 武器升級 3 英雄×3 = **14** → 隨機抽 3 張。  
- 滿編 5 人後：個人 3 + 武器 15 = **18** → 抽 3。

**質變判定**：**幾乎沒有**。

- 招募是唯一「構圖質變」（多一個自動砲台）。  
- 其餘全部是 **+傷 / -CD / +數量或半徑** 的線性堆疊。  
- 沒有：進化、變異、彈道改變、狀態異常、光環、犧牲血換傷、暴擊體系、稀有度、互斥 build 等。

**XP 曲線**（`add_xp`）：

- 起始 `xp_required = 12`  
- 升級後：`round(prev * 1.25 + 5)`  
- 允許溢出連升，但**一次只開一個升級 UI**（暫停競態處理正確）。

### 1.4 敵人種類與 AI

**三種**（`enemy_spawner.ENEMY_CONFIGS`）：

| id | 解鎖時間 | HP | 速 | 傷 | XP/金 | 權重 |
|----|----------|----|----|----|-------|------|
| normal | 0s | 20 | 88 | 8 | 2/1 | 1.0 |
| fast | 18s | 13 | 142 | 6 | 2/1 | 0.42 |
| tank | 42s | 58 | 54 | 14 | 6/3 | 0.24 |

**AI 單一模板**（`enemy.gd`）：找最近存活英雄 → 直線追 → 進距離就按 CD 造成傷害。  
**沒有**：遠程、衝刺、繞後、分裂、召喚、護盾、攔截投射物、區域危害、精英詞綴、Boss 階段。

### 1.5 生存曲線／難度縮放

僅靠 **生成密度＋敵種解鎖**，**無時間成長的 HP/傷害倍率**：

```text
spawn_count = 1 + int(elapsed / 45)
spawn_timer = max(0.22, 1.05 - elapsed * 0.006)
max_enemies = 150
```

| 時間 | 約略節奏 |
|------|----------|
| 0:00 | 每 ~1.05s 刷 1 隻，全 normal |
| 0:18 | fast 進池 |
| 0:42 | tank 進池 |
| 0:45 | 每次刷 2 隻 |
| ~2:18 | spawn_timer 觸底 0.22s |
| 2:15+ | 每次刷 3+ 隻，很快貼 150 cap |

**曲線問題預判（設計推導，非實測）**：

1. **早期偏軟**：開局就有 3 英雄 3 武器自動清場，學習曲線短。  
2. **中期靠密度**：「更多同質怪」而非「新威脅」。  
3. **後期 cap 平台**：150 滿後難度不再靠數量上升；玩家持續升級 → **DPS 雪球 vs 定點壓力**，容易變成無高潮的長尾或突然被包夾秒隊長。  
4. **無里程碑**：沒有 5/10/15 分 Boss、事件、勝利結算——時間只是 HUD 數字。

### 1.6 掉落／經濟

| 資源 | 來源 | 用途 |
|------|------|------|
| XP 寶石 | 擊殺 | 升級三選一（真成長） |
| 金幣 | 擊殺 | **僅 HUD／結算顯示**；`add_gold` 無消費 API |
| — | — | **無商店、無局內消費、無局間兌換** |

掉落 cap（R3 後）：視覺達 cap 時 **直接 grant XP/金幣**，資源不丟（見 `entity_factory.spawn_xp_gem/spawn_gold_coin`）。

### 1.7 Boss／事件／勝負

| 系統 | 狀態 |
|------|------|
| Boss | **無** |
| 精英／事件 | **無** |
| 勝場條件 | **無**（只有隊長死亡 → 任務失敗） |
| 局長 | 開放式生存至死；`reload_current_scene` 重開 |
| 隊友死亡 | 永久移出＋`dead_ids` 不可再招；**不結束遊戲** |

### 1.8 Meta 進程

| 項目 | 狀態 |
|------|------|
| 存檔／解鎖 | **無** |
| 局間升級樹 | **無** |
| 開局編制選擇 | **無**（永遠 default_squad） |
| 成就／每日挑戰 | **無** |
| 金幣跨局 | **無**（局末顯示後歸零） |

### 1.9 輸入／UI／重玩閉環

- 桌機 WASD；觸控虛擬搖桿；P/Esc 暫停。  
- 升級三選一、Game Over 摘要（時間／擊殺／金幣／等級）→ 重新開始。  
- **沒有**：build 摘要、本局高光、新解鎖提示、「差一點就…」的目標錨點。

## (2) 對抗性設計審查：淺／悶／不平衡

### 2.1 升級：線性數值海，缺 build 質變（核心痛點）

**現象**：三選一表面選項多（14–18），但**資訊熵低**——多數卡只是「誰的武器 +傷/-CD/+數量」。

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 無質變升級 | **致命（樂趣）** | 選完不改變操作手感與清場語彙，只改變數字 |
| 死選項：星環「冷卻」 | **高** | `orbit_blades.cooldown=0` 且 `cooldown_upgrade_multiplier=1.0`，卻仍進池；抽到＝白卡 |
| 池稀釋 | **高** | 人越多，**招募質變卡越少**、**同質武器卡越多**；後期三選一更無聊 |
| 個人升級只打隊長 | **中** | 隊友 HP/移速永不長；「裂隙護甲」無法救被集火的護衛 |
| 無上限／無進化 | **高** | 無限 +傷 鼓勵無腦疊 DPS；與 150 cap 疊加後雪球單調 |
| 無互斥／流派 | **高** | 不存在「這局我玩爆炸流 vs 雷鏈流」的身份感——你**同時擁有**開局三武器 |

**對抗結論**：當前升級系統比較像 **「自動射擊的數值刷機」**，不像 Roguelite 的 **「每張卡重寫規則」**。對標 Vampire Survivors / 20 Minutes Till Dawn / Brotato 的最低門檻——至少需要 **少量改變規則的升級**（分裂、軌道改變、死亡爆炸、吸怪、護盾、反傷等）。

### 2.2 敵人：三種皮，一種腦

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| AI 同構 | **致命** | 只改 HP/速/傷 = 調參怪，不是設計怪 |
| 無遠程／區域壓 | **高** | 玩家永遠「風箏即可」；沒有逼站位或走位通道 |
| 無精英讀條 | **高** | 缺少「先集火這隻」的瞬間決策 |
| 無時間 HP 縮放 | **中** | 後期只靠密度；清場快就永遠安全，清場慢就突然窒息——兩極都悶 |
| 認最近英雄 | **中（可利用）** | 隊友是仇恨海綿；但玩家幾乎無工具主動「丟仇恨／救隊友」 |

**對抗結論**：敵人目前服務「壓力條」而非「戰術詞彙」。第二十分鐘的威脅敘事仍是「紅點／黃點／紫點朝你跑」。

### 2.3 中後期失衡風險

**太簡單路徑**（高機率）：

- 開局三武器覆蓋近／鏈／線；再招爆炸與第二直線 → 五自動砲。  
- 無限疊傷／發數／穿透；150 cap 被秒清。  
- 隊長 0.65s 無敵幀 + 只需自己活 → 隊友當肉盾，體驗像看自動戰鬥。

**突然爆難路徑**：

- 隊友死光（`dead_ids`）後 DPS 斷崖，仍無回復／重招。  
- 被多方向刷出包夾；無 dash／擊退／清場大招。  
- 個人升級若一直抽到拾取範圍，生存力不長。

**缺少中間帶**：沒有「危險但可讀」的精英節奏，難度曲線是 **密度斜坡 + 玩家雪球**，不是設計過的心跳。

### 2.4 「這局不一樣」的隨機性與抉擇張力

| Roguelite 要素 | 現況 | 判定 |
|----------------|------|------|
| 開局差異 | 固定小隊 | 差 |
| 中局分支 | 招誰、疊誰的槍 | 尚可（前 2–3 次招募） |
| 稀有高影響事件 | 無 | 差 |
| 空間／地圖變異 | 空曠無限場 | 差（只有刷怪方位 RNG） |
| 風險報酬選擇 | 無寶箱／祭壇／商店 | 差 |
| 失敗有資訊 | 只有四個數字 | 弱 |

**真正有張力的抉擇目前只有**：

1. 早期：先招工匠（AOE）還是修補者（第二直線）還是先疊主力 DPS？  
2. 隊友瀕死前是否還能用隊長走位「擋／不擋」。  

其餘多數升級是 **弱偏好**，不是 **痛苦的取捨**。

### 2.5 局間回饋：「為什麼再開？」

| 動機 | 現況 |
|------|------|
| 解鎖新東西 | 無 |
| 變強再戰 | 無 Meta |
| 新策略 | 策略空間極窄 |
| 追高分／時間 | 只有裸數字，無分段目標 |
| 敘事／角色養成 | 無 |

**對抗結論**：重開按鈕存在，但 **心理帳戶是空的**。玩家結束一局後的內在獨白較可能是「又是同樣三個人朝怪跑」，而不是「下次我改走雷爆流／留金幣換復活」。

### 2.6 金幣是設計謊言

HUD 與結算強調金幣，但 **全 codebase 無 sink**。  
這會造成：

- 認知負擔（以為有經濟）→ 失望；  
- 或忽略金幣 → 擊殺掉落的一半回饋無效。  

**要嘛做店，要嘛降級為純分數並改文案**——現狀最傷信任。

### 2.7 小隊系統的潛力未兌現

已有：編隊跟隨、招募、分武器、隊友可死。  
未有：指令、陣型切換、技能連攜、被動光環、復活、隊長犧牲機制。  

目前小隊更像 **「三個綁在一起的 VS 角色」**，還不是 **「可操作的小队 fantasy」**。這是本專案相對一般幸存者的差異化機會，但尚未變成樂趣來源。

### 2.8 樂趣現況：什麼是好的？

必須公正記錄 **已成立的爽點**，否則路線圖會誤砍：

1. **自動武器手感骨架完整**：線／環／爆／鏈四語彙已齊，足以支撐質變擴充。  
2. **小隊自動輸出** 降低操作門檻，適合 Web／手機。  
3. **升級暫停三選一** 節奏正確（有呼吸感）。  
4. **技術債主幹（池、spatial、cap 解耦）** 讓「加敵加特效」有擴充餘地。  
5. **招募** 是現成的最大高潮——應被更多「準招募級」選擇包圍，而不是淹沒在 +4 傷害卡裡。

## (3) 程式品質與風險（承接 R1–R3）

### 3.1 R1–R3 契約現況（靜態）

| 項目 | 狀態 | 依據 |
|------|------|------|
| `spawn_token` 命中表 | **維持** | `enemy.get_hit_token`；projectile / orbit 用 token |
| `NodePool` double-release / `_in_pool` | **維持** | `node_pool.gd` 有拒絕邏輯 |
| 爆炸 cap 不丟傷害 | **已修（R3 回應後）** | `spawn_explosion` 先 `_apply_explosion_damage`，VFX 可 null |
| 掉落 cap 不丟資源 | **已修** | cap / acquire 失敗 → `_grant_xp_direct` / `_grant_gold_direct` |
| 熱路徑 group 掃敵 | **維持遠離武器** | 武器走 spatial；敵人 AI 仍掃 heroes（小隊規模小，可接受） |
| 壓測誠實度 | **R3 後仍可能 p95&lt;60** | CODEX_RESPONSE_R3：`STRESS_PERF_BELOW_60=true`，avg~92fps |

### 3.2 本輪仍可見的 Bug／風險

#### Bug／設計缺陷（會傷體驗）

| ID | 等級 | 問題 | 位置 | 說明 |
|----|------|------|------|------|
| D1 | **P0 體驗** | 星環「冷卻」升級無效 | `orbit_blades.tres` + `squad_manager._append_weapon_upgrade_options` | 仍進三選一，浪費稀有抉擇 |
| D2 | **P0 體驗** | 金幣無用途 | `game_manager.add_gold` 等 | 經濟系統半成品 |
| D3 | **P1** | 被動欄位全假 | `hero_data.passive_*` | 文案承諾與實作落差（若 UI 未來展示會誤導） |
| D4 | **P1** | 個人升級只作用隊長 | `squad_manager.apply_upgrade` → `leader.apply_personal_upgrade` | 隊伍成長不對稱，且無說明 |
| D5 | **P2** | 隊友與隊長共用部分 hero sprite | `pulse_artificer`/`line_mender` 重用 guardian/scout 圖 | 可讀性下降 |

#### 正確性／擴充地雷

| ID | 等級 | 問題 | 位置 | 說明 |
|----|------|------|------|------|
| C1 | **P1** | spatial 查詢不濾 `is_active` | `enemy_spatial_index.find_nearest` / `get_enemies_in_radius` | 正常路徑死亡會立刻 `unregister`；若未來延遲 unregister 或暫存「假死」會鎖死炮口／空砍。防禦式濾 active 更穩 |
| C2 | **P1** | 連鎖用 `instance_id` 當 used 集合 | `chain_lightning_weapon._cast_chain` | **單次施法內 OK**；與 R2 認屍不同域。若未來「延遲跳電」跨幀需改 token |
| C3 | **P1** | 武器／爆炸 stats 每發 `duplicate` Dictionary | `to_*_stats`、orbit `configure_orbit` | Web 單執行緒 GC 壓力；高射速多英雄時可能掉幀 |
| C4 | **P2** | `hero._apply_hero_data` 每次 `CircleShape2D.new()` | `hero.gd` | 非熱路徑；重開可接受 |
| C5 | **P2** | 決定性／seed | `arena.randomize()` | 無法複現精彩局或做每日挑戰 |
| C6 | **P2** | 投射物池耗盡靜默 skip | `NodePool.acquire` → null | 射擊武器「空槍」無反饋；擴充射速前需監控 `exhausted_count` |

#### 效能（Web 單執行緒）

| ID | 等級 | 熱點 | 說明 |
|----|------|------|------|
| P1 | **P1** | 每敵每幀找最近英雄 | `enemy._find_nearest_hero` 掃小隊；150 敵 × 最多 5 英雄尚輕，但別再 group 掃全敵 |
| P2 | **P1** | 每 pickup 找 collector | `pickup._find_collector`；180 cap 時每幀掃描成員——成員少尚可，加磁吸全域需小心 |
| P3 | **P1** | 多英雄多武器 × spatial query | 5 人全自動武器時 query 數線性上升；與 R3 壓測 p95 偏低一致 |
| P4 | **P2** | VFX merge 掃描 `active_damage_numbers` | cap 64，可接受 |
| P5 | **P2** | `explosion` 純視覺仍佔 live | 傷害已解耦；OK |

#### 決定性與測試

- 有 `PoolContract` / `GameplayCap` / `WeaponSmoke` / `StressTest` 等 debug 場景——**工程文化加分**。  
- 缺：**平衡用 headless 模擬**（存活時間分布、升級選擇分布、DPS 曲線）。下一版調曲線應補「可重現的 mock run」。

### 3.3 不應回頭破壞的紅線（給實作下一版的人）

1. **命中表必須用 `spawn_token`，禁止只靠 `instance_id` 跨生命週期。**  
2. **視覺 cap 不得吞玩法結果**（爆炸傷、XP、金幣已示範正確模式）。  
3. **武器熱路徑禁止 `get_nodes_in_group("enemies")`。**  
4. **Web 單執行緒：禁止為了爽感無上限 instantiate；擴內容先吃池與 query budget。**

## (4) 下一版更新路線圖（可落地）

務實原則：佔位美術可接受；**優先改規則與節奏**；每項控制在 Web 可承受的 entity 預算內。

### P0 — 立刻提升樂趣與平衡的快贏

#### P0-1　質變升級包「裂隙偏壓」（建議 5 張，先做 3 也可）

| 名稱 | 效果（建議） | 預期體驗 | 實作接點 | 平衡注意 |
|------|--------------|----------|----------|----------|
| **裂線分叉** | 直線彈命中後 ±20° 再裂 1 發（或固定 50% 機率） | 「彈幕變胖」的即時爽感 | 新 upgrade id → `WeaponData` flag 或 `linear_bullet_weapon` / `projectile.gd` | 限 1–2 層；裂片傷害 ×0.5、不遞迴 |
| **星環共鳴** | 環繞刃命中使目標短暫易傷 +20% 或減速 | 近身圈有戰術意義 | `orbit_projectile` 命中回調；敵加 debuff 欄位 | debuff 用 timer 字典，死亡清掉；勿每幀 new 物件 |
| **脈衝餘燼** | 爆炸中心留 1.2s 燃燒區（低 DPS） | 控場、走位誘導 | `EntityFactory` 池化 `HazardZone` 或重用 explosion 的第二階段 | **hazard 必進池**；同時 live ≤ 8 |
| **雷鏈過載** | 跳到第 N 隻時小爆（半径半於 pulse） | 清群高潮 | `chain_lightning_weapon` 末跳呼叫已有爆炸傷害 API | 與爆炸傷共用 `_apply_explosion_damage`，VFX 可省 |
| **磁暴回收** | 擊殺吸附近 XP（短距瞬吸）或擴大僅 XP 磁鐵 | 減少撿屍疲勞、中期節奏 | `enemy._die` 或 `GameManager` 旗標 + pickup | 不要全圖瞬吸（毀走位） |

**共通實作建議**：

- 在 `WeaponData` 或 runtime 加 `modifiers: Dictionary` / bool flags，避免為每張卡複製整把武器。  
- `squad_manager._append_weapon_upgrade_options` **分兩類權重**：數值卡權重低、質變卡權重高且限層。  
- **立刻移除或替換** 星環無效冷卻卡（D1）。

#### P0-2　新敵行為 2–3 種（同 scene，config 驅動）

延續現有 `Enemy` + `type_id` + config，**不要先做複雜狀態機框架**。

| 名稱 | 行為 | 預期體驗 | 實作接點 | 平衡注意 |
|------|------|----------|----------|----------|
| **裂縫射手 `ranged`** | 停在距離外射慢彈（池化 projectile，友善傷或敵彈 group） | 逼走位，不能只轉圈 | `enemy.gd` 加 behavior 分支；敵彈可用現有 `Projectile` 反色 + `can_hit` 英雄 | 彈速慢、預示 0.3s；同時敵彈 cap |
| **裂行者 `dasher`** | 蓄力後短衝刺 | 讀條→閃 | `enemy` 狀態：windup／dash／recover | windup 變色即可（無新美術） |
| **寄生囊 `spawner`**（可選） | 低血或死亡吐 2 小怪 | 優先集火目標 | `_die` 或 HP 閾值 spawn | 小怪算入 150 cap；防死亡連鎖爆池 |

**最低可做 2 種：`ranged` + `dasher`**，`min_time` 建議 30s / 55s，權重低於 normal。

#### P0-3　精英＋一個迷你 Boss 節奏

| 名稱 | 內容 | 預期體驗 | 實作接點 | 平衡注意 |
|------|------|----------|----------|----------|
| **裂隙畸變（精英）** | 每 45–60s 在刷怪中保證 1 隻：×3 HP、×1.3 傷、體型大、掉 1 個**保證質變升級選項**或大 XP | 心跳與目標 | `enemy_spawner` 計時器；config `is_elite` | 精英死亡必掉可見物；不要靜默 grant 以免無感 |
| **守門者·帷幕 proto-Boss** | 3:00 或 5:00 固定事件：高 HP、兩階段（50% 血召喚 dasher 或環形彈） | 「這局有故事」 | 新 `boss_spawner` 或 spawner 事件表；可用 tank sprite 放大 | Boss 期間降普通刷怪密度，避免鏡頭外被秒；Web 上 Boss 技能少而讀得懂 |

**勝場可選快贏**：撐過 Boss 給「階段勝利」UI，可選繼續無盡——比純死才結束更有鉤子。

#### P0-4　生存曲線調校（低成本高回報）

| 調校 | 建議 | 接點 | 注意 |
|------|------|------|------|
| 時間倍率 | 自 1:30 起敵 `max_hp`、`damage` ×(1+0.04/min) | `enemy_spawner` spawn 時寫入 config | 與玩家期望 DPS 曲線一起調；先保守 |
| 刷怪 | cap 維持 150；`spawn_count` 成長略緩，改由精英補壓 | 同上 | 避免無腦堆實體 |
| 開局 | 可選：第 0–20s 只出 normal，並略降開場隊傷 **或** 略增 early HP | 武器 tres / 敵 config | 目標是 30 秒內有一次「差點被摸到」 |
| 升級權重 | 數值卡權重 1、質變 3、招募 4；同 id 已滿層剔除 | `GameManager._build_upgrade_choices` | **最便宜的手感升級** |
| 隊長死亡 | 暫維持；但死亡摘要加「撐過時間／是否擊殺精英」 | `game_over_screen` | 給重開目標 |

## (7) 總結

Crackveil Vanguard 作為 **Web 可玩的 squad survivors 骨架** 是成功的：自動武器四語彙、小隊、池化與 spatial、升級暫停、觸控，都已達「正式可玩原型」門檻。  

但若問題是 **「作為要讓人想一直玩的 Roguelite，現在缺什麼」**，答案很尖銳：

1. **缺規則改變**（質變升級／進化）——現在幾乎只有數值。  
2. **缺威脅改變**（敵 AI／精英／Boss）——現在幾乎只有密度。  
3. **缺進程閉環**（金幣用途／Meta／開局差異）——現在局間是虛無。  
4. **缺高潮結構**（時間錨點與勝場）——現在只有死才結束。  

技術上 R1–R3 換來的穩定性，正好夠用在 **P0 內容節奏更新**；不應再優先做底層大翻修。下一版的勝負手是：**讓每一局的三選一與每一分鐘的刷怪，都講得出「不一樣的故事」**——即使用佔位方塊美術，玩家也會為了那張金色裂線分叉再按一次「重新開始」。
